# Plain Earnings Review

## 何时使用本 SKILL

当用户提供**上市公司的财报文件**（PDF / 公告文本 / 链接）并要求：
- "帮我分析一下这家公司的财报"
- "做一份经营分析报告"
- "用通俗的方式解读 XX 财报"
- "看看这家公司的现金流/订单情况"

不要在以下情况使用：
- 用户只想了解某一个具体科目（直接回答即可，不必走全流程）
- 用户要做定量估值/盈利预测（这不是本 SKILL 的目标）
- 用户提供的不是财报而是研报或新闻稿

## 致谢

本 SKILL 的分析框架源于一套实用的财务验证方法，核心思想是"利润可以调节，但现金流动和资产堆积难造假"。

## 核心理念

**新手看利润表，高手看现金流量表和资产负债表。** 利润可以通过会计手段合法调节，但现金的回流和资产的堆积更难造假。本 SKILL 的所有分析都围绕一个目的：**用现金和资产的真实变化，验证利润表讲的故事是不是真的**。

## 工作流（必须严格按顺序执行）

### Step 1：准备数据（绝对不能跳过）

1. **提取原文**：
   - **PDF**：使用 `pdfplumber` 提取所有页面文本。注意：财务报表表格列密集，抽取时易发生**列错位**（数字串到错误的列、负号被吞等），必须读整页原始文本核对上下文
   - **DOCX**：直接用 Read 工具读取内容
   - **Markdown / 纯文本**：直接读取全文
   - **网页 / 公告链接**：使用 WebFetch 提取正文
2. **建立数据信源**：所有正文出现的具体数字必须来自财报原文，不能从研报/新闻/记忆中拼凑
3. **找对比报告**：如果是季报/半年报，尽量找到上年同期的报表，以及最近一期年报，作为同比和环比的对照基准

> ⚠️ **绝对禁止**：在任何报表数据没核到原文前，凭推测/常识填数字。如果某项数据查不到，宁可在报告里写 "_待补_" 也不能编造。这条规则没有例外。

### Step 2：阅读 framework.md，理解6问题链条

读取 `framework.md`，掌握 6 个核心问题的指标定义、计算方式、行业警戒线和解读方式。

### Step 3：阅读 industry-adaptations.md，做场景适配

读取 `industry-adaptations.md`：
- 判断目标公司类型（消费 / 制造 / 科技 / 平台 / 重资产 / 轻资产）
- 判断当前盈利状态（盈利 / 亏损 / 转型期 / 周期底部）
- 选择合适的适配模板，避免机械套用

### Step 4：阅读 data-rules.md，确定数据使用规范

读取 `data-rules.md`，确认：
- 同比 vs 环比的选用原则
- 单位换算规则（元 → 万元 → 亿元）
- 关键比率的行业警戒线

### Step 5：阅读 lessons-learned.md，避开已知陷阱

读取 `lessons-learned.md`，了解过往报告中踩过的坑（特别是 PDF 抽取陷阱、推测填数陷阱、二手数据传播错误等）。

### Step 6：按 report-template.md 输出报告

读取 `report-template.md`，按模板填充内容。**重要**：模板里的章节顺序、红绿灯表、ASCII闭环图都是经过用户验证的最佳实践，不要自由发挥结构。

### Step 7：交叉核验（最后一步，不能省）

报告写完后**必须**：
1. 把正文中所有出现的具体数字逐一回查财报原文
2. 把所有比率（毛利率、收现率、有息负债率等）的计算过程在心里走一遍
3. 检查"同比/环比"的标注是否与数据来源一致
4. 检查"红绿灯"信号判断是否与正文叙述一致
5. 检查 ASCII 闭环图与表格里的数字是否一致

## 关键约束

### ✅ 必须做

1. **正文所有数字必须可在财报原文中找到出处** —— 不可用"约""大概"模糊数字，要么写精确值，要么不写
2. **首屏必须是结论**（红绿灯表 + 一句话定性），而不是数据表
3. **6问题链条必须完整出现**，不能因为"看起来不重要"就跳过某一题；信号弱的题要明确说"信号弱"
4. **每个核心数字至少与另外一个数字交叉验证一次**（如收入↑+应收↓+合同负债↑形成印证）
5. **风险章节按严重性排序**，只保留 ★★★★ 及以上进入正文
6. **报告末尾保留数据底稿附录**，给想深挖的读者留出口

### ❌ 禁止做

1. **不要在开头堆砌数据表**（违反"无财务背景也能读懂"的核心定位）
2. **不要逐表细拆三张报表**（会让读者迷失，把细节放进附录）
3. **不要因为某家公司是亏损就强行套用"为什么在亏"模板** —— 盈利公司应改为"利润质量如何"
4. **不要在正文反复点名引用方法论出处**（已在开篇致谢中交代过）
5. **不要凭推测编数字**（参考 Step 1 警告）
6. **不要写"投资建议"**，这是经营分析报告，不是研报

## 输出格式

- 默认输出 Markdown 文件，文件名格式：`{公司名}{报告期}经营分析报告_速读版.md`
- 写完后询问用户是否需要导出为其他格式（PDF / DOCX），按用户需要执行
- 保存在用户指定的工作目录下
- 不要在对话里复述报告全文，用一句话告知文件路径即可

## 处理用户反馈

用户可能提出的常见调整方向：
- "增加 XX 指标" → 先判断该指标属于 6 问题链条中的哪一题，能合并则合并，不能合并则在对应题下增加
- "数据不对" → 立即用财报原文核对，承认并修正错误，不要辩护
- "结构调整" → 在不破坏 6 问题骨架的前提下尽量满足
- "更通俗 / 更专业" → 调整用词，但保持框架不变
